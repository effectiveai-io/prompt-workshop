# prompt-workshop

업무 프롬프트를 자동으로 개선하는 실험입니다.

## 개선 대상

`prompts/` 디렉터리에 있는 마크다운 파일이 개선 대상입니다.
각 파일은 특정 업무(코드 리뷰, TC 설계, 기획서 리뷰 등)를 AI에게 시키기 위한 지침서입니다.

## 평가 기준

`evaluation/` 디렉터리에 같은 이름의 평가 기준 파일이 있습니다.
각 평가 기준은 5개 항목, 항목당 0~2점, 만점 10점입니다.

평가 기준은 **고정**입니다. 수정하지 마세요.

## 테스트 시나리오

`samples/` 디렉터리에 실습용 샘플이 있습니다.
프롬프트를 테스트할 때 이 샘플을 입력으로 사용합니다.

## 개선 사이클

사이클을 시작하면 다음을 반복합니다:

1. **현재 프롬프트 읽기**: 지정된 `prompts/*.md` 파일을 읽습니다.
2. **테스트 실행**: 그 프롬프트를 사용해서 `samples/`의 샘플에 대한 결과물을 생성합니다.
3. **자체 채점**: `evaluation/`의 평가 기준으로 결과물을 채점합니다. 각 항목별 점수와 총점을 기록합니다.
4. **개선안 도출**: 점수가 낮은 항목을 분석하고, 프롬프트의 어떤 부분을 어떻게 고치면 점수가 올라갈지 구체적으로 제안합니다.
5. **개선된 프롬프트 생성**: 제안을 반영한 새 버전의 프롬프트를 작성합니다.
6. **다음 사이클로**: 개선된 프롬프트로 2번부터 다시 시작합니다.

## 보고 형식

매 사이클마다 아래 형식으로 보고합니다:

```
## Cycle N

### 프롬프트 (vN)
(이번 사이클에서 사용한 프롬프트 전문)

### 결과물
(프롬프트를 적용해서 생성한 결과물)

### 채점
| 항목 | 점수 | 이유 |
|------|------|------|
| ... | .../2 | ... |
| **총점** | **__.../10** | |

### 개선 포인트
- 이번에 점수가 낮았던 것: ...
- 다음 버전에서 고칠 것: ...
```

## 개선 방향 원칙

- **구체적인 것이 추상적인 것보다 낫다**: "좋은 리뷰를 해줘" (X) → "변수명이 camelCase인지 확인해줘" (O)
- **적은 규칙이 많은 규칙보다 낫다**: 20개의 약한 규칙보다 5개의 강한 규칙이 더 잘 작동합니다.
- **예시가 있으면 더 좋다**: "이런 경우 이렇게 리뷰해줘"라는 예시가 포함되면 품질이 올라갑니다.
- **도메인 맥락이 가장 큰 점수 상승을 만든다**: 일반적인 코딩 규칙보다 "우리 팀/서비스만의 규칙"이 점수를 크게 올립니다.

## 주의사항

- 프롬프트가 200줄을 초과하지 않도록 합니다. 너무 길면 AI가 규칙을 무시하기 시작합니다.
- 매 사이클마다 한두 가지만 바꿉니다. 한 번에 많이 바꾸면 뭐가 효과적이었는지 알 수 없습니다.
- 점수가 같거나 떨어지면 이전 버전으로 돌아갑니다.
