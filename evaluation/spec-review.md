# 기획서 리뷰 평가 기준

10개 항목, 각 1~5점. 만점 50점.

## 채점 기준

- **1점**: 언급 없음 또는 완전히 일반적
- **2점**: 언급은 있지만 구체성 없음
- **3점**: 도메인 맥락이 일부 반영됨
- **4점**: 구체적이고 실행 가능한 수준
- **5점**: 전문가 수준의 완성도

## 평가 항목

### 1. 예외 케이스 발견 (1~5)
- 1: 기획서에 적힌 내용만 확인
- 2: "예외 상황을 고려하세요" 수준
- 3: 명백한 예외 1~2개 지적
- 4: 입력 오류, 동시성, 권한, 타이밍 등 체계적 도출
- 5: 체계적 도출 + 각 예외의 발생 확률/영향도 분석

### 2. 기술 실현 가능성 (1~5)
- 1: 기술 관점 없이 기획만 리뷰
- 2: "구현이 어려울 수 있다" 수준
- 3: 기술적 제약 1~2개 언급
- 4: 현재 아키텍처 기반 구체적 제약 + 대안 제시
- 5: 제약 + 대안 + 공수 예측 + 의존성 분석

### 3. 모호한 표현 지적 (1~5)
- 1: 모호한 표현을 그대로 통과
- 2: "좀 더 구체적으로" 수준의 일반적 피드백
- 3: "적절히", "필요시" 같은 표현 일부 지적
- 4: 모호한 표현을 모두 찾아내고 구체화 요청
- 5: 모호 표현 지적 + 구체화 예시 제안 ("적절히" → "3초 이내")

### 4. 기존 정책 정합성 (1~5)
- 1: 새 기획만 독립적으로 리뷰
- 2: "기존 기능과 충돌이 없는지 확인하세요" 수준
- 3: 관련 기존 기능 1~2개 언급
- 4: 기존 정책/기능과의 충돌 가능성을 구체적으로 지적
- 5: 충돌 지적 + 해결 방안 제안 + 영향 범위 분석

### 5. 사용자 시나리오 (1~5)
- 1: 사용자 관점 없음
- 2: "사용자 경험을 고려하세요" 수준
- 3: 주요 사용자 시나리오 1~2개 언급
- 4: 다양한 사용자 유형별 시나리오 검토
- 5: 시나리오 + 사용자 여정 맵 + 이탈 포인트 분석

### 6. 데이터 모델 검토 (1~5)
- 1: 데이터 관련 언급 없음
- 2: "데이터 구조를 정의하세요" 수준
- 3: 필요한 데이터 항목 일부 언급
- 4: 데이터 흐름 + 저장 구조 + 마이그레이션 필요성 검토
- 5: 데이터 모델 + 인덱싱 + 쿼리 성능 + 확장성 검토

### 7. 비기능 요구사항 (1~5)
- 1: 기능만 리뷰, 비기능 언급 없음
- 2: "성능을 고려하세요" 수준
- 3: 응답 시간, 동시 사용자 등 일부 언급
- 4: 성능, 확장성, 가용성, 모니터링 항목 체계적 검토
- 5: 비기능 요구사항 + 수치 기준 제안 + 측정 방법

### 8. 일정/리소스 타당성 (1~5)
- 1: 일정 관련 언급 없음
- 2: "일정이 빠듯해 보인다" 수준
- 3: 일정 위험 1~2개 지적
- 4: 기능별 공수 예측 + 일정 타당성 분석
- 5: 공수 + 리스크 + 단계별 릴리스 제안

### 9. 질문 형태 피드백 (1~5)
- 1: "~해야 합니다" 식의 일방적 지적만
- 2: 일부 질문 포함
- 3: 주요 지적을 질문 형태로 전환
- 4: 모든 지적을 "~한 경우는 어떻게?" 질문 형태로
- 5: 질문 + 맥락 설명 + 기획자가 판단할 수 있는 선택지 제시

### 10. 리뷰 구조화 (1~5)
- 1: 지적 사항을 순서 없이 나열
- 2: 대략적 분류
- 3: 영역별 분류 ([기능]/[기술]/[UX] 등)
- 4: 분류 + 심각도 + 우선순위
- 5: 분류 + 심각도 + 요약 테이블 + "이것만 해결하면 출시 가능" 판단

## 예상 점수대

| 프롬프트 수준 | 예상 점수 | 설명 |
|---|---|---|
| 빈 프롬프트 | 10~15점 | 표면적 리뷰만 |
| 기본 기준 포함 | 18~24점 | 예외 케이스, 모호 표현 지적 |
| 도메인 맥락 포함 | 25~32점 | 기술, 정책, 데이터 반영 |
| 전문가 수준 | 33~40점 | 비기능, 일정, 구조화 |
| 만점 50점 | 사실상 불가능 | 항상 개선 여지 있음 |
