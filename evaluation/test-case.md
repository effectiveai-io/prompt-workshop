# TC 설계 평가 기준

10개 항목, 각 1~5점. 만점 50점.

## 채점 기준

- **1점**: 언급 없음 또는 완전히 일반적
- **2점**: 언급은 있지만 구체성 없음
- **3점**: 도메인 맥락이 일부 반영됨
- **4점**: 구체적이고 실행 가능한 수준
- **5점**: 전문가 수준의 완성도

## 평가 항목

### 1. 정상 시나리오 (1~5)
- 1: 정상 시나리오 없음
- 2: "정상 동작 확인" 수준의 모호한 TC
- 3: 주요 정상 흐름 1~2개 포함
- 4: 다양한 정상 입력 조합 커버
- 5: 정상 흐름 + 각 단계의 상태 변화까지 검증

### 2. 비정상 시나리오 (1~5)
- 1: 비정상 시나리오 없음
- 2: "에러 케이스 확인" 수준
- 3: 명백한 에러 1~2개 포함
- 4: 입력 오류, 권한 부족, 타임아웃 등 체계적
- 5: 비정상 + 복합 실패 시나리오 (예: 동시 요청 + 잔액 부족)

### 3. 경계값 테스트 (1~5)
- 1: 경계값 테스트 없음
- 2: "최대/최소 확인" 수준
- 3: 주요 경계값 1~2개 포함
- 4: 0, 최대값, 음수, 빈 문자열 등 체계적
- 5: 경계값 + 경계값 직전/직후 + 오버플로우 시나리오

### 4. 우선순위 분류 (1~5)
- 1: 우선순위 없이 나열
- 2: 일부 TC에만 "중요" 표시
- 3: P1/P2/P3 3단계 분류
- 4: 분류 + 각 우선순위의 기준이 명확
- 5: 분류 + 기준 + "왜 이 TC가 P1인지" 도메인 근거

### 5. 사전 조건 (1~5)
- 1: 사전 조건 없음
- 2: "로그인 상태" 수준
- 3: 필요한 데이터 상태 일부 명시
- 4: 사용자 상태, 데이터 상태, 시스템 상태 모두 명시
- 5: 사전 조건 + 사전 조건 설정 방법(SQL/API) 포함

### 6. 테스트 단계 구체성 (1~5)
- 1: "기능을 테스트한다" 수준
- 2: 대략적 순서만 있음
- 3: 번호 매긴 단계 + 입력값 명시
- 4: 단계 + 입력값 + 각 단계의 기대 반응
- 5: 단계 + 입력값 + 기대 반응 + 스크린샷 위치 표시

### 7. 기대 결과 명확성 (1~5)
- 1: 기대 결과 없음
- 2: "정상 동작해야 함" 수준
- 3: 기대 결과가 있지만 검증 방법 불명확
- 4: 구체적 기대값 + 검증 방법 (UI 확인, DB 확인 등)
- 5: 기대값 + 검증 방법 + 실패 시 예상 증상

### 8. 보안/규제 시나리오 (1~5)
- 1: 보안/규제 관련 TC 없음
- 2: "보안 확인" 수준
- 3: 인증/인가 기본 TC 포함
- 4: 도메인 규제 (출금 한도, 본인 인증 등) 반영
- 5: 규제 TC + 우회 시나리오 + 감사 로그 확인

### 9. 데이터 무결성 (1~5)
- 1: 데이터 검증 TC 없음
- 2: "데이터가 정확해야 함" 수준
- 3: 기본 CRUD 후 데이터 확인
- 4: 트랜잭션 롤백, 부분 실패 시 데이터 상태 검증
- 5: 무결성 + 동시성 데이터 충돌 + 복구 시나리오

### 10. 형식 일관성 (1~5)
- 1: TC마다 형식이 제각각
- 2: 대략적 구조는 있지만 일관성 없음
- 3: TC ID, 우선순위, 단계가 대부분 포함
- 4: 모든 TC가 동일한 템플릿 사용
- 5: 템플릿 + 자동화 가능한 형식 (예: 테스트 프레임워크 연동 가능)

## 예상 점수대

| 프롬프트 수준 | 예상 점수 | 설명 |
|---|---|---|
| 빈 프롬프트 | 10~15점 | 기본적인 TC만 |
| 기본 형식 포함 | 18~24점 | 구조화된 TC |
| 도메인 맥락 포함 | 25~32점 | 보안/규제 TC 반영 |
| 전문가 수준 | 33~40점 | 경계값, 동시성, 데이터 무결성 |
| 만점 50점 | 사실상 불가능 | 항상 개선 여지 있음 |
